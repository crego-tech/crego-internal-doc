## Overview

Crego v2.0.0 is the foundational major release of the platform, delivering a complete end-to-end Loan Management System (LMS) with multi-tenant architecture. This release establishes the core infrastructure for loan origination, disbursement, repayment, demand management, co-lending, and reporting — all supported by a configurable workflow engine and a modern React-based frontend.

---
## 1. Loan Lifecycle Management
### 1.1 Account (Loan) Creation & Configuration

The platform now supports the full loan creation lifecycle from product definition to account activation. Lenders can define **Products** (e.g., Term Loan, SCF, Invoice Financing) that serve as templates, then create **Programs** under those products with specific configurations, and finally onboard individual loan **Accounts** under programs.

Each loan account carries a **Configuration** object that controls how schedules are generated, how interest is computed, how repayment is processed, and how DPD (Days Past Due) thresholds are applied. Configurations are inherited from the product/program level but can be overridden at the account level for flexibility.

  

Key capabilities delivered:
- Product, Program, and Account hierarchy with configurable inheritance

- Sub-program support with parent co-lending config inheritance

- Loan configuration schemas for Term Loan, SCF (Supply Chain Finance), and Upfront Interest modes

- Schedule generation with configurable frequency (monthly), EMI calculations, pre-EMI count handling, and rounding configuration

- Bank details validation during loan creation to ensure settlement accounts are set up before disbursement

- Co-borrower and guarantor support — multiple contacts can be linked to a single loan account with specific relationship roles

- Account status management (Draft, Active, Closed) with automatic status transitions on full settlement

  

### 1.2 Disbursement & Payout Processing

  

Once a loan is approved and activated, the disbursement module handles the release of funds. Disbursement is tightly integrated with the workflow engine and supports both simple single-disbursement and complex co-lending scenarios.

  

- Disbursement workflow graph with configurable steps (limit check, payout creation, ledger updates)

- Payout creation and processing with status tracking (Pending, Processed, Rejected)

- Deduction handling during disbursement — processing fees, insurance, and other charges can be automatically deducted from the disbursement amount before payout

- Split amount calculations for co-lending scenarios where multiple lenders share the disbursement

- Group account creation during disbursement for shared/co-lending accounts

- Limit validation before disbursement — ensures the disbursement amount does not exceed the sanctioned limit

- Ref ID generation with configurable patterns per namespace

  

### 1.3 Repayment Processing

  

The repayment engine handles incoming payments and applies them to outstanding demands in a configurable order. It supports various settlement scenarios including partial payments, excess payments, and tax settlement.

  

- Payment creation with reference ID and UTR tracking

- Configurable settlement order — lenders can define the priority in which components (principal, interest, penal, fees) are settled

- Tax calculations during settlement — GST and other applicable taxes are automatically computed and settled alongside the principal components

- Excess payment detection — when a payment exceeds the total outstanding, the excess is tracked at the account level for future adjustment or refund

- Waiver processing — specific demand components can be waived partially or fully, with audit logging of the waiver details

- Transaction splitting for co-lending — repayment amounts are automatically split between lenders based on their configured share ratios

- Account auto-closure on full settlement — when all outstanding demands are settled, the account status automatically transitions to Closed

  

### 1.4 Schedule Generation

  

The schedule engine generates repayment schedules based on the loan configuration. Schedules determine when demands are raised and how much is due on each date.

  

- EMI schedule generation with configurable tenure, frequency, and start date

- Pre-EMI schedule handling — separate pre-EMI periods before regular EMI begins

- Upfront interest schedules — interest deducted at disbursement with subvention day calculations

- Demand expression framework — a flexible expression-based system that defines how demand amounts are calculated for each schedule, supporting principal, interest, and custom component combinations

- Schedule restructuring — ability to modify schedules mid-tenure for changes in interest rate or tenure

- Include/exclude start date configuration in schedule generation

- Tax-inclusive amount calculations for demands

## 2. Demand & DPD Engine

  

### 2.1 Demand Management

  

Demands represent the amounts due from a borrower on specific dates. The demand engine automatically generates demands from the schedule and tracks their lifecycle from creation through settlement.

  

- Automated demand generation from schedule with component-wise breakdown (principal, interest, penal interest, fees)

- Demand outstanding tracking — real-time updates as payments are applied

- Component-wise demand summary — API to view aggregate outstanding across all demands, broken down by component (principal outstanding, interest outstanding, etc.)

- Tax-waived functionality — ability to waive tax on specific demand components

- Contact-level demand filtering — view demands specific to a borrower contact across their accounts

- Program-level demand filtering — aggregate demand views across all accounts in a program

- Demand status calculation — automatic status determination (Upcoming, Due, Overdue, Settled) based on due date and payment status

  

### 2.2 DPD (Days Past Due) & NPA Classification

  

The DPD system tracks how many days a demand has remained unpaid past its due date. This is critical for NPA (Non-Performing Asset) classification and regulatory reporting.

  

- DPD calculation based on demand due date vs. current date

- Peak DPD tracking — records the maximum DPD reached during the loan lifecycle

- NPA threshold configuration — lenders can define DPD thresholds that trigger NPA classification at different asset category levels (Substandard, Doubtful, Loss)

- DPD visible in transaction summary view for quick portfolio assessment

- Configurable DPD rules per product/program — different products can have different NPA classification criteria

## 3. Contact & KYC Management

  

Contacts represent borrowers, co-borrowers, guarantors, and other parties involved in the lending process. The contact module provides comprehensive identity management with deduplication and data protection capabilities.

  

- Contact creation with comprehensive profile fields (name, mobile, email, PAN, address)

- PAN-based deduplication — the system validates PAN numbers across the program hierarchy and prevents duplicate contact creation for the same PAN

- Unique contact validation across the program hierarchy — ensures a borrower is not onboarded twice under the same program tree

- Address deduplication with configurable validation rules

- Contact-Account relationship management — supports linking contacts as borrower, co-borrower, or guarantor with relation-specific metadata

- Contact Relations API with expandable serializers for viewing nested relationships

- **Props (Properties) system** — a flexible key-value property system that allows attaching custom metadata to contacts (and other entities). Props can be configured per product/program with specific data types, validation rules, and display settings

- Sensitive field masking — configurable masking rules that hide sensitive information (PAN, Aadhaar, bank details) in API responses based on user permissions. Fields are displayed as masked values (e.g., "XXXX1234") unless the user has explicit permission to view the full value

- Props encryption — sensitive props can be encrypted at rest with a configurable encryption key, adding a second layer of protection beyond masking

- Soft deletion — contacts are never permanently deleted; they are marked as inactive to preserve audit trail and historical references

## 4. Co-Lending Module

  

Co-lending enables multiple financial institutions to jointly fund a single loan, with each lender's share tracked separately throughout the lifecycle. This is essential for partnerships between banks and NBFCs under RBI's co-lending model.

  

- Partner configuration at the program level — define lending partners with their share percentages

- Partner type support — Vendor, Dealer, and other custom partner types for different co-lending arrangements

- Share ratio configuration — define how principal, interest, and other components are split between partners

- Shared account creation — when a loan is disbursed in a co-lending arrangement, the system automatically creates linked "shared" accounts for each lending partner, mirroring the main account's schedule

- Transaction splitting — repayments, disbursements, and other transactions are automatically split between the main account and shared accounts based on configured share ratios

- Parent program partner inheritance — sub-programs automatically inherit partner configurations from their parent program

- Partner uniqueness validation — prevents duplicate partner entries across the program hierarchy

- BPI (Broken Period Interest) settlement logic for co-lending — handles the interest accrued between disbursement date and the first EMI date, split according to partner shares

- Shared document functionality — documents uploaded to the main account are accessible from shared accounts

## 5. General Ledger (GL) & Accounting Module

  

The GL module provides double-entry bookkeeping capabilities integrated directly into the lending platform. Every financial transaction (disbursement, repayment, accrual, write-off) automatically generates corresponding GL entries.

  

- Double-entry GL posting with configurable chart of accounts

- Posting batch processing — transactions are grouped into batches for atomic GL posting with rollback support

- Accrual handling — interest accrual calculations that generate GL entries for accrued-but-not-yet-due interest, essential for regulatory accounting standards (IndAS/IFRS)

- Trial Balance report generation — aggregated debit and credit balances across all GL accounts for a given period

- Upfront interest accrual — special handling for loans where interest is collected upfront but needs to be recognized over the loan tenure

- Ledger expansion in APIs — schedule and transaction APIs include expandable ledger details showing the GL impact of each entry

- Outstanding calculation with accrual support — APIs that return the true outstanding amount including accrued (but not yet demanded) interest

## 6. Workflow Engine (crego-flow)

  

The workflow engine is a standalone microservice that powers all business process automation in the platform. Workflows define the sequence of steps that execute when key events occur (e.g., loan disbursement, repayment processing).

  

### 6.1 Flow Execution

  

- **Stage-Group-Node architecture** — workflows are organized into Stages (sequential phases), Groups (parallel execution units), and Nodes (individual actions). This allows complex business processes to be modeled with both sequential and parallel execution paths.

- **Node types** — Task Node (calls an API or performs a computation), API Request Node (makes external HTTP calls with configurable timeout, headers, and payload), Loop Node (iterates over a collection and executes child nodes for each item)

- **Runner system** — each execution of a workflow creates a Runner instance that tracks the execution state, logs, and outcome. Runners can be filtered by status, searched, and inspected for debugging.

- **Execution modes** — synchronous (blocking) and asynchronous (via Celery task queue) execution support

- **Activity logging** — every node execution is logged with input, output, duration, and any errors for complete audit trail

  

### 6.2 Policy Engine

  

- Policy CRUD — create, update, and delete policies that define business rules

- Policy execution API — evaluate a policy against a given input and return the decision

- JSON logic-based conditions — policies use JSON Logic expressions that can evaluate complex conditions against loan data

- Product type and time-based filters — policies can be scoped to specific product types and time windows

  

### 6.3 Template System

  

- Jinja-based templates — define dynamic content templates for notifications, documents, and API payloads

- Filter functions — built-in and custom filter functions including JWT encode/decode, parser support for transforming data within templates

- Template nodes in workflows — integrate templates into workflow execution for dynamic content generation

  

### 6.4 Omni Integration

  

- Connector APIs — the workflow engine connects to crego-omni for account creation, status updates, and data retrieval during workflow execution

- Callback URL support — external systems can trigger workflow steps via callback APIs

- Tenant-aware API calls — all requests from crego-flow to crego-omni include the correct tenant context headers

  

### 6.5 Security

  

- Secret management — store and retrieve sensitive credentials (API keys, tokens) used by workflow nodes, with per-tenant isolation

- Secret rotation script — automated rotation of encryption keys for stored secrets

- DB-alias scoped access — secrets are isolated by database alias to prevent cross-tenant access

## 7. Reports & Analytics

  

The platform includes a configurable report framework that supports both pre-built system reports and custom report generation.

  

- **Report framework** — REST API for triggering, queuing, and retrieving report outputs. Reports are generated asynchronously via Celery workers and stored for download.

- **Customer portfolio report** — comprehensive view of all loans with outstanding amounts, DPD status, and asset classification

- **Interest computation report** — detailed interest calculation breakdown for each account, used for auditing and reconciliation

- **Collection and disbursement reports** — track money inflow (repayments) and outflow (disbursements) across programs

- **Upfront interest report** — specialized report for loans with upfront interest, showing the subvention breakdown and accrual status

- **Report re-trigger** — ability to re-run a report with the same parameters to get updated results

- **Generated report storage** — reports are stored as files (Excel/PDF) and can be downloaded via API

## 8. Review & Approval System

  

The review system implements a maker-checker pattern where sensitive actions (disbursement, payout processing, configuration changes) require approval from authorized users before execution.

  

- Review creation on entity changes — when a maker creates or modifies a sensitive entity, a review request is automatically generated

- Approval/rejection workflow — checkers can approve or reject pending reviews with comments

- Review status tracking — Pending, Approved, Rejected with timestamps and reviewer details

- Props audit logging — all changes to entity properties are recorded in the audit log with before/after values

- Masking in review views — sensitive fields are masked in the review interface, with full values only visible to users with appropriate permissions

- Bulk action support for reviews — approve or reject multiple review requests at once

## 10. Multi-Tenant Architecture & Infrastructure

  

### 10.1 Tenant Isolation

  

- Every API request is scoped to a specific tenant via header-based resolution (`X-Tenant-Alias`)

- Database routing ensures each tenant's data is stored and queried in isolation

- Celery task queues are tenant-specific to prevent cross-tenant task interference

- Secret storage is scoped by database alias for credential isolation

  

### 10.2 Permissions & Licensing

  

- Role-based access control with configurable group permissions

- Predefined groups (Admin, Operations, Customer) with post-migration auto-creation

- License endpoint mapping — API access is controlled per module (LMS, GL, Reports, etc.) through license configuration

- Customer-type user support — borrowers can access a limited view of their own loan data via the customer portal

- OIDC integration — user authentication via OpenID Connect with email-based user lookup

  

### 10.3 Infrastructure

  

- Python/Django backend with Celery for async task processing, Celery Beat for scheduled tasks

- Gunicorn + Uvicorn workers for high-concurrency API serving

- Docker containerization with ECR-based image management

- GitHub Actions CI/CD with parallel Docker builds

- Slack webhook integration for deployment notifications

## Known Limitations in v2.0.0

  

- Invoice and Purchase Order financing is not yet supported (delivered in v2.1.0)

- Multi-tranche disbursement is limited to single-disbursement-per-account (enhanced in v2.1.0)

- Loan write-off and foreclosure workflows are not yet available (delivered in v2.1.0)

- Excess payment refund requires manual processing (automated in v2.1.0)

- Weekly and custom frequency schedules are not supported (added in v2.1.0)

- Audit report with analytics is not available (delivered in v2.2.0)

- Partner/customer self-service login is not supported (delivered in v2.2.0)
