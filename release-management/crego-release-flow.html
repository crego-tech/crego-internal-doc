Semantic Versioning Rules
ChangeBumpExample
Bug fix onlyPATCH2.2.0 → 2.2.1
New feature (backward compat)MINOR2.2.0 → 2.3.0
Breaking API changeMAJOR2.2.0 → 3.0.0
Hotfix (emergency patch)PATCH2.2.0 → 2.2.1
Key Principles
✓No v prefix in version strings (only in Git tags)
✓Always include 8-char commit SHA for traceability
✓Hyphens as separators between all components
✓Same format in Docker tag, K8s label, and Sentry release
✓Git tag determines all downstream version strings
✗RC tags (v2.2.0-rc.1) blocked from production deploy
Version Tracking — Git Tags as Source of Truth
All release versions are tracked via Git tags. No separate version file is needed.
Query latest internal release
git tag -l 'v*' --sort=-version:refname | grep -v '-' | head -1
Query all releases for a specific client
git tag -l '*-bcpl' --sort=-version:refname
List all client releases across all clients
git tag -l 'v*-*' --sort=-version:refname

O Release Orchestrator
One command, all repos. The Release Orchestrator (release-orchestrator.yaml) runs from crego-infra and coordinates releases across crego-omni, crego-flow, and crego-web. It replaces running manual git commands in each repo separately.
Internal Release Flow (via Orchestrator)
cutbranches from develop

QA + UATon preprod
→
shipPRs → master/main
→
tagat merge commit
→
merge-backprod → develop
StepActionWhat It DoesTriggered Automatically?
1 cut Creates release/vX.Y.Z branches from develop in all 3 repos via GitHub API No — manual trigger
2 (automatic) Branch push triggers infra-deployment.yml → builds images → pushes to GCR → ArgoCD deploys to preprod Yes — on push to release/**
3 (manual QA) QA + UAT test on preprod environment. Bug fixes go to release branch, backported to develop. No — human process
4 ship Creates PRs from release/vX.Y.Z → master/main in all repos. PRs need 2 reviewer approvals. No — manual trigger
5 tag Verifies release PR was merged, then creates tag vX.Y.Z at the merge commit SHA (not HEAD). Tag triggers prod deploy. No — manual trigger
6 merge-back Creates PRs from master/main → develop to sync release fixes back. No — manual trigger
Commands (run in sequence after each step completes)
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0 -f action=cut -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0 -f action=ship -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0 -f action=tag -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0 -f action=merge-back -f dry_run=false
Client Release Flow (via Orchestrator)
cutbranches from develop
→
client-buildpreprod → client ECR
→
Client UATon their cloud

client-buildprod → client ECR
StepActionWhat It DoesKey Difference from Internal
1 cut Creates release/vX.Y.Z-bcpl branches from develop in all 3 repos Branch name includes client suffix. Does not trigger internal infra-deployment (skip guard).
2 client-build Reads clients.yaml for client config (AWS role, region, Sentry DSN). Triggers client_release.yaml in each repo with --ref release/vX.Y.Z-bcpl. Builds Docker images and pushes to client's ECR. Uses clients.yaml (not GitHub env vars). Pushes to client AWS ECR (not GCR). No ArgoCD deploy.
3 (manual UAT) Client tests on their own infrastructure using the images pushed to their ECR. Client manages their own deployment. Fixes go to release branch, re-run client-build to redeploy.
4 tag Creates vX.Y.Z-bcpl tag in all 3 repos. Marks code as approved for client production. Tag only — no PR to master/main. Client branches never merge to our production.
5 client-build Same as step 2 but with env=prod. Builds from the vX.Y.Z-bcpl tag. Pushes prod images to client ECR. Builds from tag (immutable), not branch. Uses --ref vX.Y.Z-bcpl.
Commands (example: bcpl client release)
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0-bcpl -f action=cut -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0-bcpl -f action=client-build -f env=preprod -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0-bcpl -f action=tag -f dry_run=false
gh workflow run release-orchestrator.yaml --repo crego-tech/crego-infra -f version=v2.5.0-bcpl -f action=client-build -f env=prod -f dry_run=false
Not used for client releases: ship and merge-back are internal-only actions. Client branches don't merge to master/main — they are tag-only. No PR to production is needed because we just push images, not deploy to our infra.
Internal vs Client — Action Comparison
ActionInternal ReleaseClient Release
cut Creates release/vX.Y.Z → auto-deploys to preprod (GCR + ArgoCD) Creates release/vX.Y.Z-client → no auto-deploy (skip guard blocks infra-deployment)
client-build Not used Reads clients.yaml → triggers client_release.yaml → builds + pushes to client ECR
ship Creates PRs: release → master/main (needs 2 reviewers) Not used — client branches don't merge to prod
tag Tags at merge commit SHA → triggers prod deploy via infra-deployment Tags release branch HEAD → used as immutable ref for client-build prod
merge-back PRs master/main → develop (sync release fixes back) Not used — backport fixes manually if needed
Safety Features
✓Dry run by default — dry_run=true unless explicitly set to false
✓Idempotent — all actions skip if already done (branch/tag/PR already exists)
✓Concurrency control — only one run per version at a time
✓Pre-flight checks — validates branch exists (ship), PR was merged (tag), client in clients.yaml (client-build)
✓Merge-back gap check — cut warns if previous release wasn't merged back to develop
✓Summary job — runs if: always(), reports pass/fail across all repos

✓Always use merge commits (not squash) for release → master
✓Always merge master back to develop after every release
✓PR title must include CRE-xxx issue ID
✓Every PR needs at least 1 approval (2 for master/release)
✓Verify git tags after every prod deploy (git tag -l 'v*')
✓Backport every release-branch fix to develop (needs-backport)
✓Hotfix branches from master, never from develop
✓Add found-in:* label on every bug
✓Verify Sentry release tracking after every deploy
NEVER DO
✗Push directly to develop or master — always use PRs
✗Push new features to a release branch — it's frozen
✗Force-push to any shared branch
✗Merge your own PR — self-review is not allowed
✗Delete the develop or master branch
✗Deploy RC tags (-rc.*) to production
✗Skip review for hotfixes — 2 reviewers still required
✗Use squash merge on release → master (loses commit history)
✗Close a release without verifying tags exist in all repos
