## Overview



Crego v2.2.0 introduces partner and customer self-service login, a comprehensive fees and charges module, and full audit reporting capabilities with analytics. This release also brings significant improvements to bulk import workflows, GL configuration for new co-lending partners (Aerem, Adani, Vayana), transaction preview functionality, and the new invoice/PO financing with group drawdown support. The frontend receives feature flags, user role management, configurable themes, CDMS integration, and system date management.

## 1. Partner & Customer Login System (CRE-3707)



**What it is:** A complete user assignment and authentication system that allows partners (co-lending institutions, channel partners) and customers (borrowers) to log into the platform with scoped access — seeing only the data relevant to them.



**Why it matters:** Until now, only internal operations staff could access the platform. Partners had to request information via email or phone, and customers had no visibility into their loan status. This feature eliminates those bottlenecks by giving partners and customers direct, controlled access.



**How it works:**



- **User assignment system** — administrators can assign users to specific programs, products, or accounts. The assignment defines what the user can see and what actions they can perform. For example, a co-lending partner might be assigned to their specific program and can view all accounts under it, while a borrower is assigned to their specific loan account(s).



- **Scoped visibility** — when a partner or customer logs in, all API responses are automatically filtered to show only the entities they are assigned to. A partner user querying the accounts API will only see accounts under their assigned programs. A customer user will only see their own accounts.



- **JWT token enhancement** — JWT tokens now include `roles`, `tenant`, and `profile` claims. These additional claims allow the frontend to render the appropriate UI experience (full admin view vs. partner dashboard vs. customer portal) without additional API calls.



- **Token refresh TTL** — token refresh now uses a configurable TTL (Time-To-Live) from database settings rather than a hardcoded value. This allows administrators to set different session durations for internal staff vs. partners vs. customers based on security requirements.



**Frontend support:**



- Updated program partner registration with filters for easier partner user management (CRE-3707)

- Partner ID returned from program partner input for linking users to specific partners

- Partner customer login hotfix for edge cases in authentication flow

## 2. Fees & Charges Module (CRE-3928)



**What it is:** A comprehensive fee and charge management system that allows lenders to configure, calculate, and collect various fees at the loan, program, contact, and account levels. This includes processing fees, document fees, stamp duty, insurance charges, and any other custom fee types.



**Why it matters:** Fee income is a significant revenue stream for lending institutions. Previously, fees were handled as part of the disbursement deduction logic with limited configurability. This module makes fee management a first-class feature with full lifecycle support — from definition to collection to reporting.



**How it works:**



- **Fee configuration hierarchy** — fees can be defined at the product level (applied to all loans under the product), program level (applied to all loans in a specific program), or account level (specific to a single loan). Lower-level configurations override higher-level ones.



- **Processing fee and document fee capping (CRE-4365)** — lenders can now set maximum caps on processing fees and document fees. If the calculated fee exceeds the cap, the capped amount is used instead.



- **Tax input in fees during restructuring (CRE-4880)** — fees applied during restructuring now support tax input, allowing correct GST computation on restructuring charges.



- **GST calculation handling** — the ChargeSummaryHandler has been refactored to improve GST calculation accuracy. GST is now correctly computed based on the fee amount after any capping is applied.



- **Loan summary integration** — the loan summary API now includes processing fee details alongside the principal, interest, and other component summaries.



- **Charge summary report (CRE-4525)** — a dedicated report that shows all fees and charges levied across the portfolio, with their collection status and outstanding amounts.



- **Deduction details in summary (CRE-4701)** — deduction details are now visible in the fee and charges summary view.



**Frontend support:**



- Fees and charges configuration UI in program and account detail pages (CRE-3928)

- Deduction details display in fee and charges summary (CRE-4701)

## 3. Audit Report & Analytics (CRE-4629)



**What it is:** An enhanced audit logging system with analytics dashboards, advanced filtering, and export capabilities. This goes beyond basic audit trails to provide a queryable, analyzable record of all system actions.



**Why it matters:** Regulatory compliance requires lending institutions to maintain comprehensive audit trails. Operations managers need to monitor who did what and when, especially for sensitive actions like disbursements, write-offs, and configuration changes.



**How it works:**



- **Enhanced audit log** — the audit log now captures richer metadata for each action including entity type, entity ID, action type, user, timestamp, and a detailed diff of what changed.



- **Analytics capabilities** — the audit log supports aggregation queries: how many transactions were processed today, which users are most active, what's the average processing time for approvals, etc.



- **Advanced filtering** — the audit report page supports filtering by date range, user, action type, entity type, and custom search queries.



- **CSV export** — the entire audit report or any filtered subset can be exported as CSV.



- **Transaction process audit (CRE-4655)** — added audit trail for schedule and demand state changes during transaction processing.
