## Overview

  

Crego v2.1.0 is a major feature release that significantly expands the platform's lending product capabilities. This version introduces invoice and PO financing with drawdown support, multi-tranche disbursement for complex funding structures, loan write-off and foreclosure workflows, and a completely overhauled accrual engine. The frontend receives extensive improvements including a new GL module interface, drag-and-drop settlement ordering, and enhanced report management.

## 1. Invoice & Purchase Order Financing with Drawdown Support

  

**What it is:** A complete financing module for Supply Chain Finance (SCF) scenarios where lenders fund against invoices or purchase orders rather than traditional term loans. Borrowers submit invoices or POs as collateral, and funds are disbursed against these documents through "drawdowns."

  

**Why it matters:** This enables lenders to offer working capital solutions to businesses by financing their receivables (invoices) or procurement (purchase orders). It opens up an entirely new asset class beyond traditional term lending.

  

**How it works:**

  

- **Invoice and PO registration** — borrowers or operations teams can register invoices and purchase orders in the system with details like amount, due date, counterparty, and supporting documents. Each invoice/PO undergoes item-level validation to ensure data completeness before it can be used for financing.

  

- **Drawdown creation** — once invoices/POs are approved, a drawdown request is raised against one or more documents. The drawdown amount can be the full invoice value or a partial amount based on the lender's financing policy (e.g., 80% of invoice value).

  

- **Item validation and update logic** — each drawdown is linked to specific invoice/PO line items. The system validates that the requested amount does not exceed the available financing limit for each item and tracks how much of each document has been drawn down.

  

- **GL integration** — drawdown transactions automatically generate the correct GL entries, including any deductions (processing fees, insurance) that are netted from the disbursement amount. The GL report generator now includes invoice-level data for accurate financial reporting.

  

- **Bulk actions** — operations teams can approve, reject, or process multiple invoices, POs, and drawdowns at once using the bulk action interface, with appropriate permission checks for each action.

  

**Frontend support:** The web application includes dedicated pages for viewing and managing invoices, purchase orders, and drawdowns. The drawdown creation flow guides users through selecting documents, specifying amounts, and reviewing the disbursement details before submission.

## 2. Multi-Tranche Disbursement

  

**What it is:** The ability to disburse a single loan in multiple installments (tranches) rather than as a lump sum. Each tranche has its own disbursement date, amount, payout, and ledger entries.

  

**Why it matters:** Many lending products — particularly construction finance, project finance, and builder loans — require staged fund release tied to project milestones. Previously, the platform only supported single-disbursement-per-account, which forced workarounds for these products.

  

**How it works:**

  

- **Tranche creation** — operations teams can create multiple disbursement tranches against a single loan account. Each tranche specifies the amount, expected disbursement date, and any tranche-specific deductions.

  

- **Tranche-level ledger tracking** — each tranche generates its own set of ledger entries, allowing the lender to see the exact financial position of each disbursement. The `_build_tranche_ledgers` function creates separate ledger records for principal, interest accrual start, and any fees associated with each tranche.

  

- **Payout creation per tranche** — each tranche triggers its own payout to the borrower or vendor. Payout rejection handling is improved so that rejecting one tranche does not affect other tranches on the same account.

  

- **Schedule impact** — when a new tranche is disbursed, the repayment schedule is recalculated to account for the additional principal. The demand engine regenerates future demands based on the updated outstanding balance.

  

- **Principal demand adjustment** — the tranche disbursement logic correctly adjusts the principal demand for partial disbursements, handling edge cases where the principal demand might be null (e.g., for the first tranche before any schedule exists).

  

**Frontend support:** A dedicated "Request Tranche" button in the transaction module allows users to initiate a tranche disbursement. The UI displays all tranches for an account with their status, amount, and disbursement date. After successful tranche disbursement, users are automatically navigated to the payouts view.

## 3. Loan Write-Off

  

**What it is:** The ability to formally write off a loan that is deemed unrecoverable. When a loan is written off, the outstanding balance is removed from the books and corresponding GL entries are created to record the loss.

  

**Why it matters:** Financial institutions are required to write off loans after a certain period of non-performance (typically after all recovery options are exhausted). The write-off process has specific accounting and regulatory implications — interest accrual must stop, the outstanding must be reversed, and the GL must reflect the loss.

  

**How it works:**

  

- **Write-off transaction** — a new transaction type "Write-Off" can be initiated against an active loan account. The system validates that the loan has not already been written off (preventing double write-offs).

  

- **Conditional accrual reversal** — when a loan is written off, the system checks for any unrealized accrued interest and creates reversal entries. This ensures that interest income that was accrued but never collected is not reflected in the P&L.

  

- **Account status update** — the account status changes directly to "Written Off" without going through the normal closure flow. This is a distinct status from "Closed" (which indicates normal repayment completion).

  

- **GL posting** — write-off transactions generate specific GL entries: debit to the write-off/loss account and credit to the loan outstanding account, effectively removing the loan from the healthy portfolio.

  

**Frontend support:** The write-off option is available in the transaction actions for eligible accounts. The UI includes a confirmation step showing the outstanding amount that will be written off and the GL impact.

## 4. Loan Foreclosure

  

**What it is:** The ability for a borrower to prepay and close a loan before the scheduled tenure end. Foreclosure involves calculating the total settlement amount (principal outstanding + accrued interest + any foreclosure charges) and processing the full payment in a single transaction.

  

**Why it matters:** Borrowers frequently want to close loans early when they have surplus funds. Lenders need to accurately calculate the total payoff amount, which is different from the simple "outstanding" because it includes interest accrued since the last demand date, any applicable penalties or foreclosure fees, and potentially a benefit for early closure.

  

**How it works:**

  

- **Foreclosure preview** — before processing, users can request a foreclosure preview that calculates the exact settlement amount including all components (principal, interest, penal, fees, taxes). This gives the borrower and operations team full visibility into the payoff amount.

  

- **Flat interest handling** — for loans with flat interest computation (common in certain SME lending products), the foreclosure calculation uses a different formula than reducing-balance loans. The system correctly computes the remaining interest for flat-rate products.

  

- **Outstanding cap** — the foreclosure amount is capped at the actual outstanding to prevent overpayment. If the calculated settlement amount exceeds the outstanding (due to rounding or timing), the system caps it to the outstanding balance.

  

- **Transaction processing** — foreclosure is processed as a specialized transaction type that settles all outstanding demands and closes the account. The transaction processor handles foreclose as a distinct type alongside repayment, disbursement, and other transaction types.

  

**Frontend support:** The foreclosure option is available in the transaction actions menu. The settlement excess payment schema is updated with a foreclosure-specific enum value. Users can preview the payoff amount before confirming the transaction.

## 5. Pre-EMI Restructuring

  

**What it is:** The ability to restructure a loan's EMI schedule during or after the pre-EMI period. The pre-EMI period is the time between disbursement and the start of regular EMI payments, during which the borrower typically pays only interest.

  

**Why it matters:** Business circumstances change — a borrower might need a longer or shorter pre-EMI period, or their financial situation might change requiring EMI adjustment. The restructuring capability allows lenders to modify the loan terms without cancelling and re-creating the loan.

  

**How it works:**

  

- **Pre-EMI count management** — the system now uses a dedicated calculation method for determining the pre-EMI schedule count, making it easier to modify the pre-EMI period independently of the regular EMI schedule.

  

- **EMI simplification** — the EMI calculation logic has been refactored to remove the pre-EMI count as a direct parameter, instead computing it separately. This makes the restructuring process cleaner and reduces the risk of calculation errors.

  

- **Schedule regeneration** — when a restructuring is applied, the system regenerates the remaining schedule based on the new parameters while preserving the history of already-settled demands.

  

**Frontend support:** The pre-EMI restructuring option and skip-pre-EMI transaction actions are available in the transaction module.

## 6. Excess Payment Processing & Refund

  

**What it is:** A comprehensive system for handling excess payments — situations where a borrower pays more than the outstanding amount — and processing refunds of those excess amounts.

  

**Why it matters:** Excess payments are common in lending operations (borrowers rounding up, processing full EMI when partial outstanding remains, or pre-closing with slight overpayment). Without proper tracking and refund capability, these amounts become operational reconciliation headaches.

  

**How it works:**

  

- **Excess detection and tracking** — when a repayment amount exceeds the total outstanding, the surplus is automatically detected and recorded as an excess amount at the account level. The `excess_amount` field on payments is now filterable for easy reporting.

  

- **Program/partner-level excess** — in co-lending scenarios, excess amounts are tracked not just at the account level but also at the program-partner level. This ensures each lending partner's share of the excess is correctly accounted for.

  

- **EOD excess settlement** — excess payments can be automatically settled during the End-of-Day (EOD) process. When new demands become due, any existing excess balance is first applied before requiring additional payment from the borrower.

  

- **Refund workflow** — for cases where excess needs to be returned to the borrower, a dedicated refund workflow creates a payout transaction from the lender to the borrower. The refund amount is validated against the available excess balance.

  

- **BPI refinement** — the BPI (Broken Period Interest) settlement logic for main and shared accounts has been refined to correctly handle excess scenarios in co-lending arrangements.

  

**Frontend support:** Excess payment and refund options are available in the payment action dialogs. The settlement schema includes product-type-specific filters for the correct excess handling behavior. The excess balance is visible in the account detail view.

## 7. Accrual Engine Overhaul

  

**What it is:** A comprehensive refactoring of the interest accrual calculation engine, introducing a DateContext abstraction and supporting new accrual modes including monthly accrual and one-time accrual.

  

**Why it matters:** Interest accrual is one of the most critical and complex calculations in a lending platform. It determines how interest income is recognized in the P&L, affects regulatory reporting, and impacts the outstanding amount shown to borrowers. The v2.0.0 accrual engine worked for basic scenarios but had limitations with complex products and edge cases.

  

**Key improvements:**

  

- **DateContext abstraction** — all accrual and repayment handlers now use a `DateContext` object that encapsulates the relevant dates (transaction date, value date, accrual-from date, accrual-to date). This eliminates date-handling bugs where different handlers were using inconsistent date references and makes the system easier to reason about and test.

  

- **Monthly accrual per component** — the system now supports accruing interest on a monthly basis for each component independently. This is critical for products where different components (e.g., base interest vs. penal interest) accrue on different schedules or bases.

  

- **One-time accrual mode** — for certain product types (e.g., short-term revolving credit), interest is accrued only once at a specific point rather than on a recurring basis. The new one-time accrual mode supports this pattern.

  

- **Upfront interest accrual improvements** — the accrual logic for upfront interest products (where interest is deducted at disbursement) has been enhanced to correctly handle subvention day calculations, demand existence checks, and accrual-to-end-date edge cases.

  

- **Month-end interest collection** — a new accrual mode specifically designed for SCF (Supply Chain Finance) products where interest is calculated and collected at month-end based on the actual outstanding during the month. This supports the "Aspire SCF" requirement for penal interest calculation at month-end.

  

- **Backdated entry handling** — the transaction service now correctly handles backdated entries in transaction summaries, ensuring that accruals calculated for past periods are correctly reflected.

  

- **Filter corrections** — several fixes to ledger query filters in the AccrualHandler ensure that accrual calculations only consider the correct set of transactions for a given period, preventing double-counting or missed entries.

## 8. Settlement & Transaction Enhancements

  

### 8.1 Configurable Settlement Order per Asset Category

  

Lenders can now define different settlement orders for different asset categories (Standard, Substandard, Doubtful, Loss). For example, a lender might settle principal first for standard assets but interest first for NPA assets to maximize recovery.

  

The settlement order is configurable via a drag-and-drop interface in the frontend, where operations teams can rearrange the component priority for each asset category.

  

### 8.2 Payment Settlement Details Endpoint

  

A new API endpoint (`/payment-settlement-details/`) provides a detailed breakdown of how a specific payment was applied across demands and components. This is essential for borrower communication and audit purposes — showing exactly which demands were settled and how much was applied to each component.

  

### 8.3 Transaction Date Separation

  

The transaction model now distinguishes between three date fields: `txn_date` (when the transaction was recorded in the system), `value_date` (the effective date for financial calculations), and `bank_date` (when the bank actually processed the transaction). This separation is critical for accurate accrual calculations and regulatory reporting where the system date and bank processing date may differ.

  

### 8.4 Financial Period Validation

  

Transaction processing now validates that the transaction date falls within a valid financial period. This prevents backdated entries into closed periods and ensures that all transactions are booked in the correct accounting period.

  

### 8.5 Service Fee Settlement

  

Dedicated handling for service fee settlement in co-lending arrangements (specifically for BCPL co-lending configuration), ensuring that service fees are correctly split and settled between lending partners.
