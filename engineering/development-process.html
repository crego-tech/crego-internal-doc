Q QA Testing Process
QA Workflow Overview

When an issue moves to QA state in Linear, QA picks it up and follows this process end-to-end before taking action on the Linear issue.

Key rule: QA must complete ALL test cases and document results in the test tracker before updating the Linear issue. Do not discuss individual test cases separately — evaluate the change as a whole.
Step 1: Pick Issue & Deploy to QA Environment
  1. Pick the issue from Linear (QA state) and assign yourself
  2. Choose an available QA environment: qa1, qa2, etc.
  3. Deploy the feature branch to the chosen QA environment
  4. Select the appropriate database to copy for testing (match the client/data context of the change)
Environment selection: Coordinate with other QA team members to avoid environment conflicts. Use qa1/qa2 based on availability.
Step 2: Write & Execute Test Cases
  1. Write comprehensive (360°) test cases covering all aspects of the change — positive flows, negative flows, edge cases, regression scenarios
  2. Document all test cases in the QA Test Tracker spreadsheet
  3. Execute each test case on the deployed QA environment
  4. Record pass/fail status and notes for every test case in the tracker
QA Test Tracker (Google Sheet)
https://docs.google.com/spreadsheets/d/15hKG5-S4knNQ_f4mm8GD2TUil6tVIS8zxbfX6dm80xg/edit?gid=948003831#gid=948003831
Step 3: Take Action on Linear

Once all test cases have been executed and documented in the tracker:

OutcomeAction
All tests passMove issue to Document state in Linear → triggers merge to develop
Any test failsMove issue back to Development state with a comment summarizing all failures (link to tracker row). Do not create separate issues for each failing test case.
Important: Do not discuss or flag individual test case results one by one. Complete the full test pass, then take a single action on the Linear issue with a consolidated summary.
QA Checklist
  • ☐ Issue assigned to you in Linear
  • ☐ QA environment selected (qa1/qa2) and DB copied
  • ☐ Feature branch deployed to QA env
  • ☐ All test cases written in the QA Test Tracker
  • ☐ All test cases executed and results recorded
  • ☐ Linear issue updated with final verdict (pass → Document / fail → Development)

Typical Feature Workflow
1. Create feature branch
git checkout develop && git pull && git checkout -b feature/cre-123-my-feature
2. Commit and push
git commit -m "[CRE-123] Add authentication flow" && git push -u origin feature/cre-123-my-feature
3. Open PR (GitHub/GitLab) with title format:
CRE-123: Add authentication flow
4. After peer review approval, notify QA
# QA tests on feature branch, then approves
5. Merge to develop (GitHub PR only)
# Merge via GitHub Pull Request — do not merge locally via CLI
Branch Protection Rules
  • master/main: Require PR, at least 1 approval, dismiss stale reviews on push
  • develop: Require PR, at least 1 approval, CI must pass
  • release/*: Restrict to RM + DevOps, require CI to pass
  • Delete head branch after merge (enabled)
  • Require branches to be up to date before merge
Merge Strategy
  • Feature/Bugfix: Squash merge (clean history, 1 commit per issue)
  • Release to Main: Merge commit (preserve release history, create annotated tag after)
  • Hotfix back to Develop: Merge commit (preserve emergency context)
Merge commits on production maintain clear release history in git log.

Version Consistency Across Services

All services deployed in the same release must use the same major.minor version.

  • PATCH can differ (e.g., v2.1.0 for omni, v2.1.3 for ai)
  • Example coordinated release: omni-api v2.1.0, ai-api v2.1.0, flow-api v2.1.0, web v2.1.0
  • Use crego-internal-docs/release-management/release-notes/v2.1.0 to document coordinated changes
This ensures clients receive compatible service APIs in the same release cycle.
Web App Version Exposure

Frontend: Build step injects __APP_VERSION__ global variable (clean semver, no SHA)

Example: In browser console: __APP_VERSION__ === "2.1.0" (not "2.1.0-a1b2c3d4")

Users and support can query app version from browser console for debugging.
H Hotfix Process
When to Hotfix
  • Critical security vulnerability — applies to production data or auth
  • Production downtime — service unavailable, errors on core path
  • Data loss / corruption — database or file system integrity
  • Critical business impact — revenue-blocking or customer-facing (use judgment)
Regular bugs, even high-priority, should use normal feature/develop/release flow. Do NOT use hotfix for convenience.
Hotfix Workflow
1. Create hotfix branch from master/main
git checkout master && git pull && git checkout -b hotfix/cre-999-critical-fix
2. Fix the bug, commit, push
git commit -m "[CRE-999] Fix critical auth bug" && git push -u origin hotfix/cre-999-critical-fix
3. Code review + testing (same as feature, on hotfix branch)
4. Merge to master/main with merge commit
git checkout master && git pull && git merge --no-ff hotfix/cre-999-critical-fix
5. Create annotated tag on master/main
git tag -a vX.Y.(Z+1) -m "Hotfix: CRE-999" && git push origin master --tags
6. CI builds image vX.Y.(Z+1)-{sha8}, pushes to Prod registry
7. RM syncs ArgoCD → pod restarts → hotfix live
8. Merge hotfix back to develop with merge commit
git checkout develop && git pull && git merge --no-ff master
✓ Hotfix complete, develop synced with production
Hotfix SLA
SeverityResponseFix TargetCommunication
P0 (Critical)5 min30 min to productionSlack #incident, then email/SMS
P1 (Major)15 min2 hours to productionSlack #incidents
Hotfixes bypass the normal release cycle. Page on-call and notify stakeholders immediately.
Post-Hotfix Checklist
  • Run post-deploy verification (see K8s & Tools tab)
  • Monitor Sentry for 1 hour (no new related errors)
  • Confirm develop is synced with master/main
  • Update Linear issue to Released state
  • Document in release notes
  • Schedule postmortem (if P0)

O Orchestration & Promotion
TL Workflow
Phase 1: Planning (Daily during dev cycle)
  • Monitor Linear for completed features merged to develop
  • Track version roadmap (e.g., v2.1.0 target date)
  • Identify blocking issues, coordinate with team
Phase 2: Cut Release (T-2 days before prod target)
  • Create release/vX.Y.Z branch from develop
  • Update changelog, version files
  • Push branch → CI builds RC image → deploys to Preprod
  • Notify QA: "Release ready for testing on preprod"
Phase 3: QA & Fixes (T-1 to T-0.5 days)
  • Monitor QA progress in Linear (issues, blockers)
  • Apply preprod fixes directly to release branch
  • New RC images auto-built on each push
  • Track preprod verification checklist
Phase 4: Ship (T-0, prod deploy time)
  • Confirm QA sign-off (Linear issue → QA state approved)
  • Merge release branch to master with --no-ff
  • Create annotated tag vX.Y.Z (auto-triggers prod build + deploy)
  • Sync master back to develop
  • Monitor Sentry, Grafana during initial rollout (first 30 min critical)
Phase 5: Post-Release
  • Run post-deploy verification checklist (see K8s & Tools tab)
  • Confirm all services healthy in prod
  • Release Linear issue to Released state
  • Communicate release summary to stakeholders
Multi-Service Coordination

Typical coordinated release (all in v2.1.0):

  • crego-omni → v2.1.0-{sha}
  • crego-flow → v2.1.0-{sha}
  • crego-web → v2.1.0-{sha}

All ship within the same release window. If one service has a bug after QA, apply fix to that repo's release branch, rebuild RC image, re-test, then all images ship together.

Coordinated versioning ensures API compatibility between services in a release.

Releases & Error Tracking

SENTRY_RELEASE env var: Must match Docker image tag for proper release grouping.

Typical flow:

  1. Docker image tagged v2.1.0-a1b2c3d4
  2. K8s deployment injected with SENTRY_RELEASE=v2.1.0-a1b2c3d4
  3. Service initializes Sentry: sentry_sdk.init(..., release=os.getenv("SENTRY_RELEASE"))
  4. All errors tagged with release, Sentry groups by release
  5. Sentry UI shows "Crash Free Sessions" per release
For kubectl verification of SENTRY_RELEASE, see K8s & Tools tab.

Frontend vs Backend Verification

Backend: Errors auto-captured by Python SDK. Check Sentry UI → Issues tab → filter by release.

Frontend (Web): Browser console shows __APP_VERSION__ = "2.1.0". Sentry DSN for web in crego-web/.env.{env}. Errors appear in Sentry → Issues, tagged with app version from JS SDK.

Post-release verification: Monitor Sentry for 30 minutes after deploy. Expected: >99% crash-free sessions, no new critical issues, error rate stable or decreasing.

Web and backend errors go to same Sentry project but identified by platform (environment tag or SDK type).
C Clients & Configuration
Multi-Tenant Release Strategy

Standard: Internal release (v2.1.0) deployed to all environments simultaneously.

Client-specific: If a client needs custom features or config not in standard release:

  • Cut release/v2.1.0-acme from develop with client-specific changes
  • Tag v2.1.0-acme when ready
  • Route via K8s namespace or cluster to tenant's namespace
  • Update Linear label: client/acme, release/v2.1.0-acme
Coordinate with DevOps for routing (namespace isolation, DNS, RBAC).
Configuration Per Environment

Dev: Loose restrictions, feature flags ON, logging DEBUG

Preprod: Production-like config, feature flags match prod target, logging INFO

Prod: Strict validation, feature flags via Vault/K8s ConfigMap, logging WARN

Client-Specific Overrides: Stored in crego-infra/overlays/{env}/client-{name}/, patched via Kustomize

Never hardcode secrets. Always use Vault or K8s Secrets. ArgoCD syncs external-secrets operator to Vault → K8s Secret.

R Rules & Constraints
MUST DO
  • Always include Linear issue ID (CRE-xxx) in branch names and PR titles. Prevents orphaned code.
  • Write clear commit messages. Format: [CRE-123] Brief description of change
  • Get peer code review before merging to develop. At least 1 approval required.
  • QA must test on feature branch before approving merge. Only then does it merge to develop.
  • Update Linear issue through entire workflow. Don't manually advance states; let auto-transitions work (set labels, link PRs).
  • Sync master/main back to develop after every release. Prevents divergence.
  • Tag every release with annotated git tag. Tag format: vX.Y.Z[-prerelease] for production, develop-{sha} for dev builds.
  • Run post-deploy verification checklist after every prod/preprod deploy. See K8s & Tools tab.
NEVER DO
  • Never force-push to develop, master, or release branches. Use regular push.
  • Never merge feature directly to master/main (except hotfix). Always go through develop → release → master.
  • Never skip CI/CD for a deploy. Always let CI build and tag images.
  • Never deploy from a local machine to production. Always trigger via git tag.
  • Never commit secrets (API keys, DB passwords, JWT keys). Use Vault or K8s Secrets.
  • Never manually update production K8s manifests. All changes via git → ArgoCD sync.
  • Never create a release branch with uncommitted changes. Always commit and push first.
  • Never mix version bumps with feature commits. Separate commit: "Bump version to v2.1.0".
Branch Protection Rules (Enforced by GitHub/GitLab)
  • master/main: Require PR approval, at least 1 approver, CI pass, no force-push
  • develop: Require PR approval, CI pass, up-to-date before merge
  • release/*: Require CI pass, restrict pushes to RM + DevOps roles
  • Feature/bugfix branches: No protection (developers can force-push)
  • Hotfix branches: Restrict to DevOps after creation

Release Validation Rules
  • Linear issue must be in QA state (or Pending Release for pre-approved releases) before RM cuts release branch.
  • Release branch tests on preprod for at least 24 hours (or until QA sign-off). No emergency ship unless critical hotfix.
  • Changelog and release notes must exist before merge to master. Auto-checked via PR template.
  • All services in coordinated release must have matching MAJOR.MINOR. PATCH can differ.
  • Post-deploy verification checklist must pass within 30 minutes of prod deploy. If any check fails, trigger rollback.
Rollback Procedures

If critical issue detected in prod:

  1. RM reverts ArgoCD image tag to previous stable version (e.g., v2.0.5 if v2.1.0 is broken)
  2. kubectl triggers rolling restart → pods pull old image
  3. Monitor Sentry (error rate should drop), Grafana (latency normalize)
  4. Once stable, create hotfix branch to address root cause
  5. Do NOT re-promote same tag; create new patch tag after fix

Rollback time: <5 minutes. Always have previous stable tag available.
