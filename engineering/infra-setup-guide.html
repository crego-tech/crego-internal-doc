Platform Overview

Crego Platform is a comprehensive financial platform consisting of backend services and frontend applications deployed with a single domain architecture.

Omni Backend Services (Containerized):

  • omni-api - REST API Service (Port 8000)
  • omni-worker - Background Task Processor
  • omni-celery-beat - Task Scheduler

Flow Services (Containerized):

  • flow-api - FastAPI Workflow Engine (Port 8000)
  • flow-worker - Background Workflow Task Processor

Frontend Applications (Containerized — Nginx):

  • omni-web - React SPA served via Nginx container (Port 8000)
  • flow-web - React SPA for workflow management served via Nginx container (Port 8000)

Monitoring Services (Containerized):

  • flower - Celery Task Monitoring UI (Port 5555)

Single Domain Architecture

The entire Crego Platform (frontend and backend) is deployed using a single domain architecture (crego[-preprod].yourdomain.com) through Kubernetes Ingress / GKE Gateway API with path-based routing. All services — including frontends — run as containerized workloads on GKE (GCP) or EKS (AWS).

Ingress Path-based Routing:

  • Frontend Containers (Nginx serving static React builds):
    • / → omni-web Service (ClusterIP:8000)
    • /flow/* → flow-web Service (ClusterIP:8000)
  • Backend APIs:
    • /api/* → omni-api Service (ClusterIP:8000)
    • /flow/api/* → flow-api Service (ClusterIP:8000)
  • Monitoring:
    • /flower/* → flower Service (ClusterIP:5555)

Architecture Benefits:

  1. Unified Access — Single domain for all application components
  2. GCP-managed TLS Certificate — Automatic certificate provisioning and renewal
  3. Simplified CORS — No cross-origin issues between frontend and backend
  4. Consistent Deployment — All services (frontend + backend) use the same K8s deployment model
  5. Single DNS Management — Simplified domain and routing configuration
  6. Environment Parity — Same architecture across dev, QA, preprod, and prod
Network Policies

The Kubernetes cluster enforces a default-deny network policy. Only explicitly allowed traffic is permitted:

  • Ingress to API/Web services (port 8000): Allowed from kube-system, ingress controller, and GCP health check ranges (35.191.0.0/16, 130.211.0.0/22)
  • Ingress to Flower (port 5555): Allowed from kube-system and ingress controller only
  • Database access (port 5432): Restricted to API, worker, beat, and flower pods
  • Egress: Allowed to DNS (53), PostgreSQL (5432), RabbitMQ (5672), Redis (6379), and external services
C Container Services Configuration
Omni API Service
  • Port: 8000
  • Health Check: GET /api/health/ (lightweight) or GET /api/health/detailed/ (comprehensive)
  • Resources (Requests / Limits): CPU: 1000m / 2000m, Memory: 2Gi / 4Gi
  • Replicas: 2–3 (HPA based on CPU 70% / Memory 80%)
Omni Worker Service
  • Purpose: Background task processing
  • Health Check: Container-level ping check
  • Resources (Requests / Limits): CPU: 500m / 2000m, Memory: 512Mi / 2Gi
  • Replicas: 1 (default), auto-scaled via KEDA up to 3 based on RabbitMQ queue depth
Omni Celery Beat Service
  • Purpose: Periodic task scheduling
  • Health Check: Database connectivity check
  • Resources (Requests / Limits): CPU: 200m / 500m, Memory: 256Mi / 1Gi
  • Replicas: 1 only (single scheduler instance required)
Flow API Service
  • Port: 8000
  • Purpose: FastAPI-based workflow execution engine with MongoDB integration
  • Health Check: GET /flow/api/health/ (lightweight) or GET /flow/api/health/detailed/ (comprehensive)
  • Resources (Requests / Limits): CPU: 1000m / 2000m, Memory: 1Gi / 2Gi
  • Replicas: 1–3 (scale based on workflow volume)
  • Dependencies: MongoDB, RabbitMQ, Omni API (for JWT authentication)

Authentication:

  • Method: JWT tokens issued by Omni service
  • Validation: Flow service validates tokens via JWKS endpoint from Omni
  • Flow: User → Omni (auth) → JWT token → Flow API (validate via JWKS)
  • Configuration: JWT_VERIFICATION_KEY must match Omni's signing key
Flow Worker Service
  • Purpose: Background workflow task processing using Celery
  • Working Directory: $PROJECT_DIR/project
  • Health Check: ps aux | grep celery (process check)
  • Resources (Requests / Limits): CPU: 1000m / 2000m, Memory: 1Gi / 2Gi
  • Replicas: 1 (default instance)
  • Dependencies: MongoDB, RabbitMQ
Omni Web Service
  • Purpose: React SPA served via Nginx container for the main platform UI
  • Port: 8000 (Nginx)
  • Path: / (root, catch-all)
  • SPA Routing: try_files $uri $uri/ /index.html
  • Health Check: GET /health (returns JSON {"status":"healthy"})
  • Security Context: Runs as user 101 (nginx), non-root
  • Resources (Requests / Limits): CPU: 50m / 200m, Memory: 64Mi / 256Mi
  • Replicas: 1–2
  • Dependencies: Omni API for backend communication
Flow Web Service
  • Purpose: React SPA served via Nginx container for workflow management
  • Port: 8000 (Nginx)
  • Path: /flow
  • SPA Routing: try_files $uri $uri/ ${endpoint_prefix}index.html
  • Runtime Env Var (required): endpoint_prefix=/flow/ — injected via envsubst at container start. Without this, flow-web will 404 on sub-routes.
  • Health Check: GET /flow/health (returns JSON {"status":"healthy"})
  • Security Context: Runs as user 101 (nginx)
  • Resources (Requests / Limits): CPU: 50m / 200m, Memory: 64Mi / 256Mi
  • Replicas: 1–2
  • Dependencies: Flow API for backend communication
Flower Service (Celery Monitoring)
  • Port: 5555
  • Purpose: Celery task monitoring and management dashboard
  • Health Check: TCP socket check on port 5555
  • Resources (Requests / Limits): CPU: 100m / 500m, Memory: 128Mi / 512Mi
  • Replicas: 1
  • Ingress Path: /flower
  • Authentication: Basic Auth via FLOWER_BASIC_AUTH env var (format: user:password)
  • Dependencies: RabbitMQ (broker connectivity)
Autoscaling Configuration

API Services: Horizontal Pod Autoscaler (HPA) based on CPU utilization (70% target) and memory utilization (80% target).

Celery Workers: KEDA ScaledObject with dual triggers:

  • RabbitMQ Queue Depth: Scales when queue length exceeds 50 messages
  • CPU Utilization: Scales at 80% CPU usage

Scaling Parameters:

  • Polling interval: 30 seconds
  • Cooldown period: 300 seconds (5 minutes)
  • Min replicas: 1, Max replicas: 3
Note: Frontend containers (omni-web, flow-web) and Flower do not autoscale — they run at fixed replica counts.

Ingress / Path-Based Routing (Critical)

All services sit behind a single domain (https://crego[-preprod].yourdomain.com) and are reached via path-based routing. Path order matters — more-specific prefixes must come before the root catch-all, otherwise omni-web will shadow the other services.

Reference Kubernetes Ingress snippet
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: crego-ingress
  annotations:
    # Raise body size to support document uploads (default is 1 MB on Nginx ingress)
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
spec:
  tls:
    - hosts: [crego.yourdomain.com]
      secretName: crego-tls
  rules:
    - host: crego.yourdomain.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend: { service: { name: omni-api, port: { number: 8000 } } }
          - path: /flow/api
            pathType: Prefix
            backend: { service: { name: flow-api, port: { number: 8000 } } }
          - path: /flow
            pathType: Prefix
            backend: { service: { name: flow-web, port: { number: 8000 } } }
          - path: /flower
            pathType: Prefix
            backend: { service: { name: flower,   port: { number: 5555 } } }
          - path: /
            pathType: Prefix
            backend: { service: { name: omni-web, port: { number: 8000 } } }
TLS Termination: TLS is terminated at the ingress / load balancer, not inside containers. All backend services receive plain HTTP on port 8000.
Ingress Body Size & Timeouts (Document Uploads)

The platform accepts document uploads which are streamed through the ingress to Omni API and then to the configured object storage bucket. Default ingress body size limits will cause uploads to fail with HTTP 413.

  • Minimum proxy-body-size: 50 MB (raise if larger documents are expected)
  • proxy-read-timeout: 120s (default 60s may cut off large uploads)
  • proxy-send-timeout: 120s
  • Upstream (Django) limit: DATA_UPLOAD_MAX_MEMORY_SIZE is set in the Omni API settings and must also accommodate the expected size

Nginx Hardening (omni-web / flow-web)

Both Nginx containers ship with production-hardened configuration. Clients do not need to modify Nginx — this is documented for security review and compliance purposes.

  • Security Headers applied at Nginx:
    • X-Frame-Options: DENY
    • X-Content-Type-Options: nosniff
    • X-XSS-Protection: 1; mode=block
    • Referrer-Policy: strict-origin-when-cross-origin
  • Static asset caching: 1-year immutable cache for hashed assets (.js, .css, .woff2, .png, .jpg, .svg, .ttf)
  • Gzip: Disabled at Nginx. Compression is performed by the ingress / load balancer after TLS termination — do not enable gzip inside the container
  • Non-root: runs as uid 101 (nginx). Compatible with restricted Pod Security Standards
  • Runtime config (flow-web only): The endpoint_prefix env var is substituted via envsubst at container start — must be set to /flow/

D Deployment Steps
Deployment Order
  1. Infrastructure Services: PostgreSQL, MongoDB, Redis, RabbitMQ
  2. Core API Services: omni-api (required for authentication across all services)
  3. Background Workers: omni-worker, omni-celery-beat, flow-worker
  4. API Services: flow-api (depends on omni-api for authentication)
  5. Frontend Containers: omni-web, flow-web (Nginx containers serving React SPAs)
Service Dependencies
  • flow-api: Requires omni-api for JWT token validation
  • flow-worker: Requires MongoDB and RabbitMQ for task processing
  • omni-worker/omni-celery-beat: Require PostgreSQL and RabbitMQ
  • All services: Share Redis for caching and session management
Step 1: Infrastructure Setup
  1. Deploy required infrastructure services:
    • Omni Services: PostgreSQL, Redis, RabbitMQ
    • Flow Services: MongoDB, RabbitMQ (shared), Redis (shared)
  2. Configure shared environment variables using your secrets management solution:
    • All environment variables are shared across services (see Environment Variables section)
    • Configure authentication and database connections
    • Set up cross-service JWT authentication
  3. Set up networking and security groups for service communication
  4. Configure load balancer routing for prefix-based service routing
Secret Management

Secrets are managed via External Secrets Operator (ESO) which syncs secrets from cloud-native secret managers into Kubernetes Secrets.

  • GCP: Google Secret Manager (Workload Identity auth)
  • AWS: AWS Secrets Manager (IAM role-based auth)

Secrets are stored as a single JSON document per environment (e.g., app-dev, app-preprod, app-prod) and auto-refreshed every 5 minutes.

Important: Never hardcode secrets in deployment manifests or ConfigMaps. All sensitive values must go through the cloud secret manager → ExternalSecret → Kubernetes Secret pipeline.
Step 2: Service Deployment

Deploy all services with the shared environment configuration. All services are stateless and can be deployed using your preferred orchestration platform (Kubernetes, ECS, Docker Swarm).

Auto-Migration: The Omni API service automatically runs database migrations (migrate) and static file collection (collectstatic) during Gunicorn startup. It uses a PostgreSQL advisory lock (ID: 123456789) to prevent concurrent migrations across multiple replicas. No manual migration step is required during deployment.
Step 3: Post-Installation Configuration

After successful deployment, configure application settings via the admin interface:

  1. Access Admin Interface: Navigate to https://crego.yourdomain.com/api/admin/
  2. Login: Use the admin credentials from environment variables (ADMIN_USERNAME, ADMIN_PASSWORD)
  3. Configure Settings: Go to Core > Settings to configure:
    • Authentication Methods: OIDC, SMS OTP, Email OTP
    • OIDC Settings: Configure identity providers (issuer, client credentials, JWKS endpoint)
    • SMS Service: Kaleyra configuration for OTP delivery
    • Email Service: SMTP configuration for notifications
    • JWT Settings: Token lifetimes and custom claims
    • Login Settings: Authentication methods per user type (staff/customer/partner)
Cross-Service Authentication (Flow ↔︎ Omni)

Flow services depend on Omni for authentication. Here's how it works:

  1. User Authentication: Users authenticate with Omni service
  2. JWT Token Issuance: Omni issues JWT tokens with shared signing key
  3. Token Validation: Flow API validates tokens using the same JWT_VERIFICATION_KEY
  4. JWKS Endpoint: Flow service can optionally fetch public keys from Omni's JWKS endpoint

Required Configuration:

  • Both services must share the same JWT_SIGNING_KEY and JWT_VERIFICATION_KEY
  • Flow service must have network access to Omni service for token validation
  • SIMPLE_JWT settings must be identical across both services

Secret Key Rotation

This management command provides comprehensive validation, corruption detection, and verification to ensure safe key rotation without data loss. Run it as part of periodic security maintenance or whenever a secret key needs to be retired.

Command Usage:

# Basic rotation with interactive confirmation
pipenv run python manage.py rotate_secret_key

# Generate and use a new key automatically
pipenv run python manage.py rotate_secret_key --new-key "$(python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())")"

# Dry run to see what would be rotated
pipenv run python manage.py rotate_secret_key --dry-run

# Skip corrupted data and continue with valid records
pipenv run python manage.py rotate_secret_key --skip-corrupted

# Create backup before rotation
pipenv run python manage.py rotate_secret_key --backup

# Detailed reporting
pipenv run python manage.py rotate_secret_key --report-only -v 2

Command Options:

OptionDescription
--dry-runShow what would be rotated without making changes
--backupCreate backup of encrypted data before rotation
--forceSkip confirmation prompts (for automation)
--new-key KEYUse specific new key (base64 encoded Fernet key)
--skip-corruptedSkip corrupted/undecryptable data and continue with valid data
--report-onlyOnly report data status without performing rotation
-v 2Verbose output showing detailed breakdown and corruption details

Startup Health Validation

The Omni API service performs a comprehensive health check during application startup (in wsgi.py). This check validates connectivity to PostgreSQL, Redis, and RabbitMQ before the application begins accepting traffic.

Critical: If any startup health check fails, the application process terminates itself via SIGTERM. This ensures unhealthy pods are immediately restarted by Kubernetes rather than serving requests with broken dependencies. Check container logs for detailed component status on startup failures.

Troubleshooting Guide

Common Issues:

  1. Database Connection Failed
    • Check DEFAULT_POSTGRES_DB_* environment variables
    • Verify database server accessibility
    • Confirm SSL configuration
  2. Celery Tasks Not Processing
    • Verify RabbitMQ connectivity (RABBITMQ_URI)
    • Check worker health: pipenv run celery -A project.celery_config:celery_app inspect ping
    • Review worker logs for task failures
  3. Health Check Failures
    • Use /health/detailed/ for component-specific status
    • Check individual service connectivity (PostgreSQL, Redis, RabbitMQ)
    • Review application logs for specific error messages
  4. MongoDB Connection Issues (Flow Service)
    • Check DEFAULT_MONGODB_URI and DEFAULT_MONGODB_NAME environment variables
    • Verify MongoDB server accessibility and authentication
    • Confirm SSL configuration for MongoDB Atlas connections
    • Check database user permissions
  5. Flow Authentication Issues
    • Verify Flow service can reach Omni API (OMNI_INTERNAL_HOST)
    • Check JWT token validation between services
    • Ensure Omni service is running and accessible
    • Review OIDC issuer configuration
  6. General Authentication Issues
    • Verify OIDC configuration if using SSO
    • Check JWT key configuration
    • Ensure admin user creation was successful
Infrastructure Requirements

1. PostgreSQL Database

  • Purpose: Primary database for application data and Celery results
  • Deployment: Amazon RDS (recommended) or self-managed
  • Version: PostgreSQL 14.x or 16.x
  • Configuration:
    • SSL required (POSTGRES_DB_SSLMODE=require)
    • Connection timeout: 10 seconds
    • Connection pooling recommended
  • Required Extensions: None (uses standard PostgreSQL features)

2. Redis/Valkey Cache

  • Purpose: Django caching layer and session storage
  • Deployment: Amazon ElastiCache or self-managed
  • Versions: Redis 6.2.6+ or Valkey 8.0+
  • Configuration:
    • Max connections: 100
    • JSON serialization
    • Connection retry on timeout

3. RabbitMQ Message Broker

  • Purpose: Celery task message broker
  • Deployment: AWS MQ (recommended) or self-managed
  • Features Required:
    • Message priorities (1-10 range)
    • Message TTL (24 hours)
    • Durable queues
    • Dead letter exchanges
  • Queue Configuration:
    • Default queue: omni-default-queue
    • Dead letter queue: omni-dead-letter-queue

4. MongoDB Database

  • Purpose: Flow service data storage for workflows, documents, and execution history
  • Deployment: MongoDB Atlas (recommended) or self-managed
  • Version: MongoDB 8.x
  • Configuration:
    • SSL/TLS enabled (mongodb+srv:// or ssl=true)
    • Connection timeout: 10 seconds
    • Connection pooling recommended
    • Authentication required for production
  • Storage Requirements:
    • Workflow definitions and execution history
    • Document storage and metadata
    • User session data (if not using Redis)
  • High Availability: Replica set configuration recommended for production

5. File Storage

  • Purpose: Document and file storage
  • Supported Backends:
    • Any S3-compatible object storage (AWS S3, GCP Cloud Storage, Azure Blob Storage, MinIO, Ceph)
  • Configuration: Managed through Django admin interface

6. Authentication

  • Default: Username/password authentication
  • Configurable Methods:
    • OIDC: OpenID Connect providers (AWS Cognito, Okta, Auth0, Azure AD)
    • OTP SMS: SMS-based one-time passwords via Kaleyra
    • OTP Email: Email-based one-time passwords
  • User Types: Staff, Customer, Partner (each with configurable login methods)
  • Token Types: JWT with HS256/RS256 algorithms

7. External Services (Optional)

  • Crego License Server: License verification and feature flag management
  • Authentication Services: External identity providers for OIDC authentication
  • File Storage Services: Document and file storage for omni services
  • Monitoring Services: Application monitoring and logging (Sentry, Remote Syslog)
