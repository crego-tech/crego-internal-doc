Security Architecture
Authentication/authorization flow, encryption mechanisms, access boundaries
A
Authentication & Authorization
Crego ↔ IDFC: OAuth2 + JWT
Client credentials grant with RS256-signed JWT client assertion. Tokens scoped per API operation and cached with TTL.
Crego ↔ DeHaat: Internal Auth
Private VPC — service-to-service authentication via internal API keys / mutual TLS. No public credential exchange.
User Access: JWT Sessions
FinOps / DC users authenticate via Crego frontend. JWT-based session tokens with role-based access control (RBAC).
E
Encryption Mechanisms
In Transit: TLS 1.2+
All external traffic encrypted. Managed certificates auto-provisioned for crego.dehaat.com. TLS terminated at load balancer.
Payload: AES-256-CBC
All IDFC API request/response payloads additionally encrypted with AES-256-CBC using IDFC-provisioned keys — encryption on top of TLS.
At Rest: Cloud-managed
Database and storage encrypted at rest using DeHaat's cloud-managed encryption keys. Secrets stored in cloud Secret Manager, synced via External Secrets Operator.
B
Access Boundaries
Network Isolation
Default-deny network policies. Pods accept traffic only from ingress controller on whitelisted ports. No direct internet access to pods.
Egress Control
Outbound restricted via NAT gateway. Only IDFC endpoints and monitoring services whitelisted. All egress flows through a single regional IP.
DeHaat Infra Boundary
All Crego services run within DeHaat's cloud account. No cross-account access. DeHaat retains full infrastructure ownership and audit control.
